## Feature Engineering

### NaN მნიშვნელობების დამუშავება

NaN მნიშვნელობების შესავსებად გამოვიყენე შემდეგი მიდგომები:

- **კატეგორიული ცვლადები** — NaN შეივსო `'None'`-ით, რადგან ბევრ შემთხვევაში NaN ნიშნავს კონკრეტული მახასიათებლის არარსებობას (მაგ. `FireplaceQu` — სახლს ბუხარი არ აქვს).
- **რიცხვითი ცვლადები** — NaN შეივსო მედიანით, რადგან საბინაო მონაცემებში დიდია გადახრა (skew) და მედიანა მდგრადია გამოყოფილი მნიშვნელობების (outlier-ების) მიმართ. საშუალოს გამოყენების შემთხვევაში გამონაკლისებს დიდი გავლენა ექნებოდა.

### გასუფთავების მიდგომები (Cleaning)

შემდეგი სვეტები წავშალეთ სხვადასხვა მიზეზის გამო:


| სვეტი                            | წაშლის მიზეზი                                                                                                                                           |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `Id`                             | მხოლოდ იდენტიფიკატორია, ფასთან კავშირი არ აქვს                                                                                                          |
| `Utilities`                      | თითქმის ყველა მნიშვნელობა `AllPub`-ია — ნულოვანი ვარიაცია                                                                                               |
| `Street`                         | 99.6% `Pave` — ნულოვანი ვარიაცია                                                                                                                        |
| `PoolQC`, `Alley`, `MiscFeature` | 94–99% ცარიელი. სახლების უმრავლესობას არ ჰქონდა აუზი. იმის გამო, რომ თითქმის ყველა მნიშვნელობა ნული იქნებოდა, ბინარულ მახასიათებლადაც კი გამოუსადეგარია |
| `Fence`                          | 80.75% missing — წავშალეთ, მაგრამ დავამატეთ `HasFence` ბინარული მახასიათებელი                                                                           |
| `MiscVal`                        | ძალიან გადახრილია და თითქმის არცერთ სახლს ჰქონდა ამ ველში მნიშვნელობა                                                                                   |


### კატეგორიული ცვლადების რიცხვებად გარდაქმნა

გამოვიყენე ორი მიდგომა:

**1. Ordinal Encoding** — ცვლადებისთვის, რომლებსაც აქვთ ბუნებრივი თანმიმდევრობა:

```python
quality_map = {'None': 0, 'Po': 1, 'Fa': 2, 'TA': 3, 'Gd': 4, 'Ex': 5}
```

ეს გამოვიყენე შემდეგ მახასიათებლებზე: `ExterQual`, `ExterCond`, `BsmtQual`, `BsmtCond`, `HeatingQC`, `KitchenQual`, `FireplaceQu`, `GarageQual`, `GarageCond`.

**2. One-Hot Encoding** — დანარჩენი კატეგორიული ცვლადებისთვის, რომლებსაც არ აქვთ ბუნებრივი თანმიმდევრობა, გამოვიყენე `OneHotEncoder(drop='first', handle_unknown='ignore')`. ამ პროცესმა მახასიათებლების რაოდენობა გაგვიზარდა, მაგრამ Feature Selection ეტაპზე ამასაც მოვუვლით.

### ახალი მახასიათებლების შექმნა

შევქმენი შემდეგი ახალი მახასიათებლები:


| მახასიათებელი  | ფორმულა                                                              | დასაბუთება                                                      |
| -------------- | -------------------------------------------------------------------- | --------------------------------------------------------------- |
| `HouseAge`     | `YrSold - YearBuilt`                                                 | სახლის ასაკი უფრო ინფორმაციულია, ვიდრე აშენების კონკრეტული წელი |
| `RemodAge`     | `YrSold - YearRemodAdd`                                              | რამდენი წლის წინ იყო ბოლო რემონტი                               |
| `GarageAge`    | `YrSold - GarageYrBlt`                                               | იგივე, გარაჟისთვის                                              |
| `TotalBath`    | `FullBath + 0.5·HalfBath + BsmtFullBath + 0.5·BsmtHalfBath`          | სველი წერტილების საერთო რაოდენობა                               |
| `TotalSF`      | `TotalBsmtSF + 1stFlrSF + 2ndFlrSF`                                  | სახლის საერთო ფართობი                                           |
| `TotalPorchSF` | `OpenPorchSF + EnclosedPorch + 3SsnPorch + ScreenPorch + WoodDeckSF` | ყველა ვერანდისა და გემბანის ჯამური ფართობი                      |
| `HasFence`     | `Fence.notna().astype(int)`                                          | ღობის არსებობა/არარსებობა                                       |


ამის შემდეგ მოვიშორეთ ის მახასიათებლები, რომლებიც ახალი ნიშნულების შესაქმნელად გამოვიყენეთ და აღარ გვჭირდება — მაგალითად, აშენებისა და გაყიდვის წლები.

## Feature Selection

### გამოყენებული მეთოდები

**1. Variance Threshold** — პირველ ეტაპზე ამოვიღეთ ნულოვანი ან თითქმის ნულოვანი ვარიაციის მქონე ცვლადები (`threshold=0.01`). ეს დაგვეხმარა One-Hot Encoding-ის შედეგად გაბერილი, ფაქტობრივად კი ერთგვაროვანი სვეტების მოშორებაში.

**2. Correlation Filter** — გამოვითვალეთ Pearson-ის კორელაცია სამიზნე ცვლადთან (`SalePrice`) და ავარჩიეთ top 50 ყველაზე მაღალი კორელაციის მქონე ცვლადი. ეს მიდგომა იძლევა `**corr_top50`** feature set-ს. კორელაცია წრფივია.

**3. Mutual Information** — გამოვიყენეთ `mutual_info_regression`, რათა გამოვავლინოთ **არაწრფივი** კავშირებიც სამიზნე ცვლადთან. ავარჩიეთ top 50 მახასიათებელი. ამ მიდგომამ მოგვცა `**mi_top50`** feature set.

**4. Combined Ranking** — გავაერთიანეთ Correlation და Mutual Information-ის რანგები და ავარჩიეთ top 60 ცვლადი. შედეგი არის `**combined_top60`** feature set — ცვლადები, რომლებიც ორივე კრიტერიუმით რელევანტურია.

### შედეგების შეფასება


| Feature Set      | ცვლადების რაოდენობა | საუკეთესო CV RMSE |
| ---------------- | ------------------- | ----------------- |
| `corr_top50`     | 50                  | 0.1432            |
| `mi_top50`       | 50                  | **0.1376** ✅      |
| `combined_top60` | 60                  | 0.1428            |


საუკეთესო შედეგი მივიღეთ `**mi_top50`** feature set-ით.

## Training

### ტესტირებული მოდელები

სულ ჩავატარეთ 20+ ექსპერიმენტი სხვადასხვა მოდელებითა და ჰიპერპარამეტრებით.

---

#### 1. Ridge Regression

Ridge Regression იყენებს L2 რეგულარიზაციას. ვცადეთ სხვადასხვა `alpha` მნიშვნელობები:


| Run                           | Alpha  | Feature Set    | CV RMSE    | Train RMSE | Gap     | შეფასება        |
| ----------------------------- | ------ | -------------- | ---------- | ---------- | ------- | --------------- |
| `ridge_a0.01_combined_top60`  | 0.01   | combined_top60 | 0.1431     | 0.1331     | +0.0100 | ოდნავ overfit   |
| `ridge_a1_combined_top60`     | 1      | combined_top60 | 0.1427     | 0.1333     | +0.0094 | კარგი           |
| `ridge_a10_combined_top60`    | 10     | combined_top60 | 0.1428     | 0.1349     | +0.0079 | კარგი           |
| `ridge_a10_mi_top50`          | 10     | mi_top50       | **0.1376** | 0.1311     | +0.0065 | ✅ საუკეთესო     |
| `ridge_a100_combined_top60`   | 100    | combined_top60 | 0.1435     | 0.1378     | +0.0057 | underfit იწყება |
| `ridge_a10000_combined_top60` | 10 000 | combined_top60 | 0.1589     | 0.1543     | +0.0046 | underfit        |


**დასკვნა:** დაბალი `alpha` (0.01) იწვევს overfitting-ს (დიდი gap train-სა და CV-ს შორის). ძალიან მაღალი `alpha` (10 000) იწვევს underfitting-ს (მაღალი შეცდომა ორივე სეტზე). ოპტიმალურია `alpha=10` `mi_top50` feature set-ზე.

---

| Run             | Alpha  | CV RMSE | Train RMSE | შეფასება        |
| --------------- | ------ | ------- | ---------- | --------------- |
| `lasso_a0.0001` | 0.0001 | 0.1428  | 0.1342     | კარგი           |

| `lasso_a0.01`   | 0.01   | 0.1513  | 0.1489     | underfit იწყება |
| `lasso_a0.1`    | 0.1    | 0.1782  | 0.1771     | underfit        |

#### 4. Random Forest

Random Forest — გადაწყვეტილების ხეების ანსამბლი (bagging). ვცადეთ სხვადასხვა სიღრმეები:


| Run                 | max_depth | n_estimators | CV RMSE | Train RMSE | Gap     | შეფასება                             |
| ------------------- | --------- | ------------ | ------- | ---------- | ------- | ------------------------------------ |
| `rf_d2_n100`        | 2         | 100          | 0.1821  | 0.1798     | +0.0023 | **underfit** — ძალიან მარტივი მოდელი |
| `rf_d5_n100`        | 5         | 100          | 0.1512  | 0.1423     | +0.0089 | underfit                             |
| `rf_d10_n100`       | 10        | 100          | 0.1389  | 0.0821     | +0.0568 | **overfit** — დიდი gap               |
| `rf_dnone_n100`     | None      | 100          | 0.1401  | 0.0312     | +0.1089 | **ძლიერი overfit**                   |
| `rf_d10_n200_leaf3` | 10        | 200          | 0.1371  | 0.0934     | +0.0437 | overfit                              |


**დასკვნა:** `max_depth=2` იწვევს underfitting-ს — მოდელი ძალიან მარტივია, ორივე სეტზე მაღალი შეცდომით. `max_depth=None` იწვევს ძლიერ overfitting-ს: train RMSE 0.03, CV RMSE კი 0.14. ეს ნათლად ჩანს overfit map გრაფიკზე.

#### 5. Gradient Boosting

Gradient Boosting ხეებს თანმიმდევრულად აგებს — თითოეული ასწორებს წინა მოდელის ნარჩენ შეცდომებს:


| Run                  | n_estimators | learning_rate | max_depth | CV RMSE | Train RMSE | შეფასება                  |
| -------------------- | ------------ | ------------- | --------- | ------- | ---------- | ------------------------- |
| `gbm_n50_lr0.1_d3`   | 50           | 0.1           | 3         | 0.1412  | 0.1187     | underfit — ძალიან ცოტა ხე |
| `gbm_n100_lr0.1_d3`  | 100          | 0.1           | 3         | 0.1389  | 0.0981     | კარგი                     |
| `gbm_n300_lr0.1_d3`  | 300          | 0.1           | 3         | 0.1371  | 0.0712     | overfit იწყება            |

**დასკვნა:** მაღალი `learning_rate` ბევრ ხეს რომ დაემატება, overfitting-ს იწვევს. დაბალი `n_estimators=50` იწვევს underfitting-ს. ოპტიმალურია `learning_rate=0.05` და `n_estimators=300`.

---

### ჰიპერპარამეტრების ოპტიმიზაციის მიდგომა

თითოეული მოდელისთვის გამოვიყენეთ **5-fold Cross-Validation** CV RMSE-ის გამოსათვლელად. ვლოგავდით:


| მეტრიკა       | აღწერა                                   |
| ------------- | ---------------------------------------- |
| `cv_rmse`     | 5-fold cross-validation RMSE (log scale) |
| `train_rmse`  | Training set RMSE                        |
| `overfit_gap` | `cv_rmse − train_rmse`                   |


მაღალი `overfit_gap` მიუთითებს overfitting-ზე. თუ ორივე RMSE მაღალია — underfitting.

### საბოლოო მოდელის შერჩევის დასაბუთება

საბოლოო მოდელად ავარჩიეთ **Ridge Regression (`alpha=10`, `mi_top50` features)**:

- Kaggle Public Score: **0.13955**

### ჩაწერილი მეტრიკების აღწერა

თითოეული ექსპერიმენტისთვის ვლოგავდით შემდეგ ინფორმაციას:

**პარამეტრები (Params):**


| პარამეტრი                    | აღწერა                                                        |
| ---------------------------- | ------------------------------------------------------------- |
| `model_family`               | მოდელის ტიპი (Ridge, Lasso, RandomForest და ა.შ.)             |
| `feature_set`                | გამოყენებული feature set-ის სახელი                            |
| `n_features`                 | feature-ების რაოდენობა                                        |
| სპეციფიკური ჰიპერპარამეტრები | `alpha`, `max_depth`, `n_estimators`, `learning_rate` და ა.შ. |


**მეტრიკები (Metrics):**


| მეტრიკა       | აღწერა                                                                        |
| ------------- | ----------------------------------------------------------------------------- |
| `cv_rmse`     | 5-fold cross-validation Root Mean Squared Error (log scale) — მთავარი მეტრიკა |
| `cv_rmse_std` | CV RMSE-ის სტანდარტული გადახრა                                                |
| `train_rmse`  | Training set-ზე RMSE                                                          |
| `overfit_gap` | `cv_rmse − train_rmse` (მაღალი = overfitting)                                 |


### საუკეთესო მოდელის შედეგები
